Architecture Overview

How the System is Built

A layered, session-authenticated platform. Admins define forms visually; the backend generates real PostgreSQL tables on publish; end users fill public forms that persist to those dynamic tables.

☕

Backend — Spring Boot 3

  • Java 21, Spring Security (session-based)
  • Spring Data JPA + Hibernate
  • JdbcTemplate for dynamic SQL
  • Table-per-form schema via SchemaManager
  • Global exception handling via @ControllerAdvice
  • Maven build, no profiles
⚡

Frontend — Next.js (App Router)

  • React 18 with App Router
  • TailwindCSS + custom design system
  • AuthContext + ThemeContext providers
  • Dynamic form renderer — no hardcoded fields
  • Client-side rule engine + visibility logic
  • DnD form builder (DnD Kit)
🐘

Database — PostgreSQL 17

  • Static tables: forms, fields, versions, users, roles
  • Dynamic tables: form_data_{code} per form
  • No Flyway / Liquibase — app-controlled DDL
  • Soft deletes on forms and submissions
  • UUID primary keys throughout
  • JSONB rules column on versions
🔄

Form Versioning

  • Each form has multiple FormVersion snapshots
  • Only one version is isActive = true at a time
  • Draft versions are editable; active is read-only
  • Activating a version drops existing user drafts
  • Every submission stores its formVersionId
⚙️

Rule Engine

  • JSON-stored conditional rules per version
  • Actions: SHOW, HIDE, REQUIRE, VALIDATION_ERROR
  • Operators: EQUALS, NOT_EQUALS, GT, LT, CONTAINS
  • AND/OR compound conditions (nested)
  • Evaluated on both frontend (live) and backend (submit)
📤

Submission Pipeline

  • Public POST /api/v1/submissions (no auth)
  • Draft save + resume per authenticated user
  • Backend validates all fields + runs rule engine
  • Data written to dynamic form_data_* table
  • Soft-delete + bulk restore on submissions
  • CSV export with injection protection

Dynamic Schema

Table-Per-Form Architecture

When a form is published, SchemaManager creates a dedicated PostgreSQL table. Each fieldKey maps to a typed column. Schema evolution adds columns but never drops them.

fieldKey Slugification

Generated by AppConstants.sanitizeKey():

  • → Trim and lowercase the label
  • → Spaces and hyphens become underscores _
  • → Strip all non-alphanumeric characters
  • → Prefix f_ if starts with a digit
"Employee Name" → employee_name
"Phone Number (*)" → phone_number
"123Score" → f_123score

Schema Evolution Rules

✅Add new columns when new fields are added
✅Existing columns preserved when fields removed
✅Nullable columns for backward compatibility
🚫Column drops never happen automatically
🚫Schema drift at publish or submit time blocks operation
🚫Reserved SQL keywords rejected as fieldKeys

Submission Pipeline

Validation & Submission Flow

Every form submission goes through a deterministic multi-stage validation pipeline before any data is written to the database.

Backend Submission Pipeline

1
Version Check

Resolve active form version. Reject if versionId in payload doesn't match the currently active version.

2
Field Mapping

Build fieldKey → FormField map from the active version. Reject any keys not in the schema.

3
Required Fields

Check all fields marked required = true have non-empty values.

4
Field-Level Validation

validateValue() runs per field: length limits, regex patterns, date/time bounds, number ranges, email/URL/phone format checks.

5
Rule Engine

RuleEngineService.validateSubmission() evaluates REQUIRE and VALIDATION_ERROR rules against submitted values.

6
Schema Drift Check

Verify the dynamic table columns match the form's field definitions before writing.

7
Write to Dynamic Table

getSafeValue() type-converts each value, then inserts/updates a row in form_data_{code}. A metadata row is written to form_submission_meta.

  • Max 100 validation rules per form

Conditional Logic

Rule Engine

Rules are stored as a JSON array in form_version.rules. Each rule has a condition tree (AND/OR nested) and an action. Rules are evaluated live in the browser and enforced server-side on submit.

Action Types

SHOW HIDE REQUIRE VALIDATION_ERROR

SHOW/HIDE control field visibility client-side. REQUIRE dynamically makes a field mandatory. VALIDATION_ERROR throws an error with a custom message when the condition is met.

Condition Operators

EQUALS NOT_EQUALS GREATER_THAN LESS_THAN CONTAINS

Numeric comparisons (GT/LT) use Double.parseDouble() for tolerance. String comparisons are case-insensitive.

Compound Conditions

Conditions can nest using logicalOperator: "AND" | "OR" with a conditions[] array. The evaluator handles unlimited depth recursively.

Example Rule JSON

{
  "condition": {
    "logicalOperator": "AND",
    "conditions": [
      {
        "field": "employment_type",
        "operator": "EQUALS",
        "value": "contract"
      },
      {
        "field": "years_experience",
        "operator": "GREATER_THAN",
        "value": "3"
      }
    ]
  },
  "action": {
    "type": "SHOW",
    "targetField": "contract_details"
  }
}

Rules are stored in form_version.rules as a JSON text array. Parsed by Jackson into List<FormRuleDTO> at evaluation time. The frontend parses the same JSON and runs an identical evaluator (evaluateConditionNode()) for live feedback.

Security Framework

Authentication & Protection

Spring Security manages session lifecycle. No JWT — server-side session only. CORS is locked to the frontend domain. One session per user enforced server-side.

Security Configuration

🔑
Session-Based Auth (No JWT)

JSESSIONID cookie. Server controls session lifecycle. sessionFixation().migrateSession() on login.

🔒
One Session Per User

maximumSessions(1). Concurrent login returns a JSON error instead of silently expiring the old session.

🌐
CORS — Strict Origin

Restricted to http://localhost:3000. allowCredentials = true required for cookie forwarding.

🛡️
Public Endpoints

POST /auth/login, POST /auth/register, GET /forms/{id}, GET /forms/{id}/rules, GET /forms/{id}/lookup/**, POST /submissions are all public. Everything else requires authentication.

🔐
Method-Level Authorization

@PreAuthorize("isAuthenticated()") on protected controller methods. Global fallback: anyRequest().authenticated().

Data Integrity & Injection Protection

🧹
fieldKey Validation

SchemaManager.validateColumnName() rejects reserved SQL keywords (SELECT, FROM, WHERE, etc.) and non-alphanumeric names before any DDL is executed.

📊
CSV Injection Protection

Any exported cell value starting with =, +, -, or @ is prefixed with a single quote to neutralize Excel formula injection.

🔢
Dynamic Query Safety

Table names are validated against [a-zA-Z0-9_]+ before use in dynamic SQL. All user values go through parameterized JDBC prepared statements.

📁
File Upload Protection

File size validated against field-specific maxFileSize (MB). File UUID is stored in the dynamic table — the actual file binary lives in a separate FileMetadata store.

🔑
Passwords

Stored using Spring Security's BCryptPasswordEncoder. Never exposed in any API response.

Access Control

Roles & Module Permissions

Access is controlled through a Role → Module mapping system. Each role is assigned a set of modules (navigation items + API access grants). The frontend sidebar and route guards read the user's module list from GET /api/v1/menu.

Role System Enum Access Scope Typical Permissions
MASTER_ADMIN SystemRole.MASTER_ADMIN Global — all users and forms Manage users, roles, modules, all forms, all submissions. Full admin panel access.
USER SystemRole.USER Personal — own forms only Create and manage own forms, view own submissions. No access to admin panel.

Module System

Modules are navigation and permission units stored in the module table. Each module has a controller, action, and prefix that map to frontend routes. The GET /api/v1/menu endpoint returns only the modules the current user's roles have access to, driving the sidebar dynamically.

  • Modules can have parent/sub-parent hierarchy for nested menus
  • Icon CSS class stored per module for UI rendering
  • Role–Module assignment via RoleModule join table

Form Ownership

Every form is owned by the user who created it via the user_id foreign key. FormService.getAllForms() scopes queries to WHERE user_id = currentUser, ensuring users only see their own forms. MASTER_ADMIN can access all forms.

  • Dashboard stats are also scoped per user
  • Submission metadata tracks submitted_by username
  • Form ownership does not change after creation
